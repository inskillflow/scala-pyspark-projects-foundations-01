## 1. C'est quoi un outil de build ?

Un **outil de build** (ou gestionnaire de build) fait 4 choses essentielles :

```
┌────────────────────────────────────────────────────────────────┐
│                     Outil de build                             │
│                                                                │
│  1. DÉPENDANCES    → télécharge Spark, Scala, etc. depuis     │
│                       Internet (Maven Central, Ivy...)         │
│                                                                │
│  2. COMPILATION    → transforme votre .scala/.java en         │
│                       bytecode JVM (.class)                    │
│                                                                │
│  3. TESTS          → exécute les tests unitaires               │
│                                                                │
│  4. PACKAGING      → crée un JAR/fat-JAR déployable           │
└────────────────────────────────────────────────────────────────┘
```

Les trois outils principaux dans l'écosystème JVM :

| Outil | Créé par | Langage config | Spécialité |
|---|---|---|---|
| **Maven** | Apache (2004) | XML (`pom.xml`) | Standard Java/entreprise |
| **SBT** | Lightbend (2008) | Scala (`build.sbt`) | Scala natif |
| **Gradle** | Gradle Inc. (2007) | Groovy/Kotlin (`build.gradle`) | Android + modern Java |

## 2. SBT — L'outil natif Scala

### Qu'est-ce que SBT ?

SBT (*Scala Build Tool*) a été conçu **par et pour** la communauté Scala.
Sa configuration est écrite **en Scala** — ce qui le rend très puissant mais parfois déroutant.

### Fichier de configuration : `build.sbt`

```scala
// build.sbt — Configuration du projet TP Spark SQL
name         := "TP-SparkSQL"
version      := "1.0"
scalaVersion := "2.12.18"

val sparkVersion = "3.5.1"

libraryDependencies ++= Seq(
  "org.apache.spark" %% "spark-core"  % sparkVersion,
  "org.apache.spark" %% "spark-sql"   % sparkVersion,
  "org.apache.spark" %% "spark-mllib" % sparkVersion
)
```

> **Notation `%%` vs `%` :**
> - `%%` ajoute automatiquement le suffixe `_2.12` (version Scala) au nom du JAR
> - `%`  utilise le nom exact sans modification
>
> Ainsi `"org.apache.spark" %% "spark-core" % "3.5.1"` devient
> `spark-core_2.12-3.5.1.jar` automatiquement.

### Structure d'un projet SBT

```
mon-projet/
├── build.sbt                  ← configuration principale
├── project/
│   ├── build.properties       ← version de SBT
│   └── plugins.sbt            ← plugins SBT
└── src/
    ├── main/
    │   ├── scala/             ← code source Scala
    │   └── resources/         ← fichiers de config, données
    └── test/
        └── scala/             ← tests ScalaTest / Specs2
```

### Commandes SBT essentielles

```bash
sbt compile          # Compiler
sbt run              # Lancer le main
sbt "runMain TP_StockSQL"   # Lancer une classe spécifique
sbt test             # Lancer les tests
sbt package          # Créer un JAR simple
sbt assembly         # Créer un fat-JAR (avec plugin sbt-assembly)
sbt clean            # Nettoyer les fichiers compilés
sbt reload           # Recharger la config build.sbt
sbt                  # Mode interactif (console SBT)
```

### Avantages de SBT

✅ **Config en Scala** — logique, conditions, réutilisation du code dans la config
✅ **Compilation incrémentale** — ne recompile que les fichiers modifiés (très rapide)
✅ **Support natif de Scala 3** — pas de configuration spéciale
✅ **Résolution `%%`** — gestion automatique des suffixes de version Scala
✅ **Mode watch** — `~compile` recompile à chaque sauvegarde
✅ **Console REPL** — `sbt console` lance un REPL Scala avec vos classes disponibles

### Inconvénients de SBT

❌ **Syntaxe complexe** — la configuration est du Scala, pas du XML lisible
❌ **Démarrage lent** — SBT démarre plus lentement que Maven
❌ **Documentation parfois insuffisante** — moins d'exemples sur Stack Overflow
❌ **Moins connu des équipes Java** — frein à l'adoption dans les entreprises Java-first

## 3. Maven — Le standard Java/entreprise

### Qu'est-ce que Maven ?

Maven est le standard de facto du monde Java depuis 2004.
Sa configuration est en **XML** (`pom.xml`) — verbeux mais universel et bien documenté.

### Fichier de configuration : `pom.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">

  <modelVersion>4.0.0</modelVersion>

  <!-- ═══════════════════════════════════════════════════
       Identité du projet
  ════════════════════════════════════════════════════ -->
  <groupId>com.monentreprise</groupId>
  <artifactId>tp-sparksql</artifactId>
  <version>1.0</version>
  <packaging>jar</packaging>
  <name>TP Spark SQL — BigData2</name>

  <!-- ═══════════════════════════════════════════════════
       Propriétés (versions centralisées)
  ════════════════════════════════════════════════════ -->
  <properties>
    <maven.compiler.source>21</maven.compiler.source>
    <maven.compiler.target>21</maven.compiler.target>
    <scala.version>2.12.18</scala.version>
    <scala.compat.version>2.12</scala.compat.version>
    <spark.version>3.5.1</spark.version>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  </properties>

  <!-- ═══════════════════════════════════════════════════
       Dépendances
  ════════════════════════════════════════════════════ -->
  <dependencies>

    <!-- Scala standard library -->
    <dependency>
      <groupId>org.scala-lang</groupId>
      <artifactId>scala-library</artifactId>
      <version>${scala.version}</version>
    </dependency>

    <!-- Spark Core -->
    <dependency>
      <groupId>org.apache.spark</groupId>
      <artifactId>spark-core_${scala.compat.version}</artifactId>
      <version>${spark.version}</version>
    </dependency>

    <!-- Spark SQL -->
    <dependency>
      <groupId>org.apache.spark</groupId>
      <artifactId>spark-sql_${scala.compat.version}</artifactId>
      <version>${spark.version}</version>
    </dependency>

    <!-- Spark MLlib -->
    <dependency>
      <groupId>org.apache.spark</groupId>
      <artifactId>spark-mllib_${scala.compat.version}</artifactId>
      <version>${spark.version}</version>
    </dependency>

    <!-- Tests : ScalaTest -->
    <dependency>
      <groupId>org.scalatest</groupId>
      <artifactId>scalatest_${scala.compat.version}</artifactId>
      <version>3.2.17</version>
      <scope>test</scope>
    </dependency>

  </dependencies>

  <!-- ═══════════════════════════════════════════════════
       Plugins de build
  ════════════════════════════════════════════════════ -->
  <build>
    <sourceDirectory>src/main/scala</sourceDirectory>
    <testSourceDirectory>src/test/scala</testSourceDirectory>

    <plugins>

      <!-- Plugin Scala : compile le code Scala -->
      <plugin>
        <groupId>net.alchim31.maven</groupId>
        <artifactId>scala-maven-plugin</artifactId>
        <version>4.9.1</version>
        <executions>
          <execution>
            <goals>
              <goal>compile</goal>
              <goal>testCompile</goal>
            </goals>
          </execution>
        </executions>
        <configuration>
          <scalaVersion>${scala.version}</scalaVersion>
          <args>
            <arg>-deprecation</arg>
            <arg>-feature</arg>
          </args>
        </configuration>
      </plugin>

      <!-- Plugin Assembly : crée un fat-JAR avec toutes les dépendances -->
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-assembly-plugin</artifactId>
        <version>3.6.0</version>
        <configuration>
          <archive>
            <manifest>
              <mainClass>TP_StockSQL</mainClass>
            </manifest>
          </archive>
          <descriptorRefs>
            <descriptorRef>jar-with-dependencies</descriptorRef>
          </descriptorRefs>
          <finalName>tp-sparksql</finalName>
        </configuration>
        <executions>
          <execution>
            <id>make-assembly</id>
            <phase>package</phase>
            <goals>
              <goal>single</goal>
            </goals>
          </execution>
        </executions>
      </plugin>

      <!-- Plugin Surefire : exécute les tests -->
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-surefire-plugin</artifactId>
        <version>3.2.2</version>
        <configuration>
          <skipTests>false</skipTests>
        </configuration>
      </plugin>

    </plugins>
  </build>

</project>
```

### Structure d'un projet Maven

```
mon-projet-maven/
├── pom.xml                    ← configuration principale (XML)
└── src/
    ├── main/
    │   ├── scala/             ← code source Scala
    │   │   └── TP_StockSQL.scala
    │   └── resources/         ← application.properties, log4j2.xml...
    └── test/
        ├── scala/             ← tests
        └── resources/
```

### Commandes Maven essentielles

```bash
mvn compile              # Compiler
mvn exec:java -Dexec.mainClass="TP_StockSQL"   # Lancer
mvn test                 # Tester
mvn package              # Créer le JAR
mvn package -DskipTests  # Créer le JAR sans tests
mvn install              # Installer dans le dépôt local (~/.m2)
mvn clean                # Nettoyer target/
mvn clean package        # Nettoyer + re-packager (le plus courant)
mvn dependency:tree      # Afficher l'arbre des dépendances
mvn versions:display-dependency-updates  # Voir les mises à jour dispo
```

### Lancer le projet Spark Maven

```bash
# Compiler et créer le fat-JAR
mvn clean package

# Lancer avec java directement
java --add-opens=java.base/sun.nio.ch=ALL-UNNAMED \
     -jar target/tp-sparksql-jar-with-dependencies.jar

# Ou via maven exec
mvn exec:java \
    -Dexec.mainClass="TP_StockSQL" \
    -Dexec.args="data/AAPL.csv" \
    -Djdk.module.illegalAccess=permit
```

### Avantages de Maven

✅ **Standard universel** — connu par tous les développeurs Java depuis 20 ans
✅ **Excellent support IDE** — IntelliJ, Eclipse, VSCode — support natif parfait
✅ **Écosystème immense** — des milliers de plugins, documentation abondante
✅ **Entreprise-friendly** — Nexus, Artifactory, CI/CD l'utilisent nativement
✅ **pom.xml lisible** — même sans connaître Maven, le XML se lit facilement
✅ **Cycle de vie standardisé** — `validate → compile → test → package → install → deploy`
✅ **Stable et prévisible** — pas de surprises dans les builds

### Inconvénients de Maven

❌ **Verbosité** — un `pom.xml` Spark fait facilement 150+ lignes vs 15 lignes en SBT
❌ **Pas natif Scala** — nécessite un plugin externe (`scala-maven-plugin`)
❌ **Pas de résolution `%%`** — il faut écrire manuellement `spark-core_2.12`
❌ **Moins flexible** — la configuration XML ne permet pas de logique conditionnelle
❌ **Builds plus lents** — pas de compilation incrémentale aussi fine que SBT

## 4. Comparaison technique détaillée

### Déclarer la même dépendance Spark

```scala
// SBT — 1 ligne
"org.apache.spark" %% "spark-core" % "3.5.1"
```

```xml
<!-- Maven — 5 lignes -->
<dependency>
  <groupId>org.apache.spark</groupId>
  <artifactId>spark-core_2.12</artifactId>
  <version>3.5.1</version>
</dependency>
```

### Tableau comparatif complet

| Critère | SBT | Maven |
|---|---|---|
| **Langage de config** | Scala (DSL) | XML |
| **Verbosité** | Concis | Verbeux |
| **Courbe d'apprentissage** | Steep (Scala requis) | Douce (XML universel) |
| **Support Scala natif** | ✅ Natif | ⚠️ Via plugin |
| **Résolution version Scala (`%%`)** | ✅ Automatique | ❌ Manuel |
| **Vitesse de compilation** | ✅ Incrémentale rapide | ❌ Recompile tout |
| **Vitesse de démarrage** | ❌ Lente (JVM SBT) | ✅ Plus rapide |
| **Support IntelliJ** | ✅ Excellent | ✅ Natif parfait |
| **Support Eclipse** | ⚠️ Partiel | ✅ Natif parfait |
| **Écosystème plugins** | Moyen | ✅ Immense |
| **Documentation** | ⚠️ Correcte | ✅ Abondante |
| **Popularité entreprise** | Scala-only | ✅ Universel |
| **CI/CD (Jenkins, GitLab)** | ✅ Supporté | ✅ Standard |
| **Dépôt d'artefacts (Nexus)** | ✅ Compatible | ✅ Natif |
| **Reproducibilité des builds** | ✅ | ✅ |
| **Multi-module** | ✅ | ✅ |
| **Scala 3 support** | ✅ Natif | ⚠️ Plugin requis |

## 5. Projet Spark avec Maven — Guide complet

### Étape 1 — Installer Maven

**Télécharger :** https://maven.apache.org/download.cgi
→ Choisir `apache-maven-3.9.6-bin.zip`

**Installer sur Windows :**
```powershell
# Extraire dans C:\tools\maven
Expand-Archive "apache-maven-3.9.6-bin.zip" -DestinationPath "C:\tools"
Rename-Item "C:\tools\apache-maven-3.9.6" "C:\tools\maven"

# Ajouter au PATH (permanent)
$currentPath = [System.Environment]::GetEnvironmentVariable("PATH", "Machine")
[System.Environment]::SetEnvironmentVariable(
    "PATH", "$currentPath;C:\tools\maven\bin", "Machine"
)

# Vérifier (nouveau terminal)
mvn --version
# Apache Maven 3.9.6 (bc0240f3c744dd6b6ec2920b3cd08dcc295161ae)
# Maven home: C:\tools\maven
# Java version: 21.0.3
```

### Étape 2 — Créer la structure du projet Maven

```powershell
# Créer la structure manuellement
New-Item -ItemType Directory -Force "C:\00-projetsGA\projet1-maven\src\main\scala"
New-Item -ItemType Directory -Force "C:\00-projetsGA\projet1-maven\src\main\resources"
New-Item -ItemType Directory -Force "C:\00-projetsGA\projet1-maven\src\test\scala"
New-Item -ItemType Directory -Force "C:\00-projetsGA\projet1-maven\data"
```

Ou via l'archétype Maven (génère automatiquement la structure) :
```bash
mvn archetype:generate \
  -DgroupId=com.bigdata2 \
  -DartifactId=tp-sparksql-maven \
  -DarchetypeArtifactId=maven-archetype-quickstart \
  -DinteractiveMode=false
```

### Étape 3 — Créer le pom.xml

Copier le `pom.xml` de la section 3 dans `C:\00-projetsGA\projet1-maven\pom.xml`.

### Étape 4 — Copier le code Scala

```
src/main/scala/TP_StockSQL.scala   ← même code que dans le projet SBT
src/main/scala/TP_Income.scala
```

### Étape 5 — Compiler et lancer

```powershell
cd C:\00-projetsGA\projet1-maven

# Compiler
mvn compile

# Lancer TP_StockSQL
mvn exec:java `
    -Dexec.mainClass="TP_StockSQL" `
    -Dexec.args="data/AAPL.csv" `
    "-Dexec.jvmArgs=--add-opens=java.base/sun.nio.ch=ALL-UNNAMED"

# Créer le fat-JAR
mvn clean package -DskipTests

# Lancer le fat-JAR
java --add-opens=java.base/sun.nio.ch=ALL-UNNAMED `
     -jar target/tp-sparksql-jar-with-dependencies.jar `
     data/AAPL.csv
```

### Étape 6 — Ouvrir dans IntelliJ

```
File → Open → sélectionner le dossier projet1-maven
IntelliJ détecte pom.xml → Import Maven project → OK
```

IntelliJ charge **nativement** les projets Maven (pas de plugin nécessaire).

## 8. Quel outil choisir selon votre situation ?

### Arbre de décision

```
Vous commencez un projet Scala/Spark...
              │
              ▼
   Votre équipe connaît-elle Java/Maven ?
         /              \
       OUI               NON
        │                 │
        ▼                 ▼
  Projet mixte ?    C'est un projet
  (Java + Scala)     100% Scala ?
    /       \          /       \
  OUI       NON      OUI       NON
   │         │        │         │
   ▼         ▼        ▼         ▼
MAVEN      MAVEN     SBT      GRADLE
(standard  (connu   (natif   (moderne,
 d'équipe)  partout)  Scala)  multi-lang)
```

### Recommandations par profil

#### Vous êtes étudiant / débutant en Scala
→ **Commencer par SBT**
- Moins de XML à écrire
- Les tutoriels Scala utilisent tous SBT
- IntelliJ l'intègre parfaitement

#### Vous travaillez dans une équipe Java
→ **Utiliser Maven**
- Tout le monde connaît `mvn clean package`
- L'intégration CI/CD (Jenkins, GitLab) est déjà configurée pour Maven
- Les revues de code seront plus faciles

#### Vous visez un poste Data Engineer en France
→ **Apprendre les deux** — mais priorité Maven
- Les grandes ESN (Capgemini, Sopra, Accenture) utilisent Maven
- Les banques et assurances sont Maven-first
- SBT est un plus, pas un prérequis

#### Vous rejoignez une startup Scala / Akka
→ **Maîtriser SBT**
- L'écosystème Scala (Akka, Play, ZIO) publie ses exemples en SBT
- Les projets open source Scala sont quasi tous en SBT

#### Vous faites du Spark sur cluster (Databricks, HDInsight)
→ **Maven pour les fat-JARs**
- `mvn clean package` crée facilement un fat-JAR soumettable avec `spark-submit`
- Meilleure intégration avec les pipelines CI/CD des clouds

## 9. Gradle — Le troisième concurrent

Gradle mérite une mention car il gagne du terrain dans les projets modernes.

```kotlin
// build.gradle.kts — même projet Spark en Gradle Kotlin DSL
plugins {
    kotlin("jvm") version "1.9.21"
    scala
    id("com.github.johnrengelman.shadow") version "8.1.1"  // fat-JAR
}

val sparkVersion = "3.5.1"
val scalaVersion = "2.12"

dependencies {
    implementation("org.apache.spark:spark-core_$scalaVersion:$sparkVersion")
    implementation("org.apache.spark:spark-sql_$scalaVersion:$sparkVersion")
    implementation("org.apache.spark:spark-mllib_$scalaVersion:$sparkVersion")
}
```

| Critère | Gradle vs SBT | Gradle vs Maven |
|---|---|---|
| Verbosité | Similaire | Beaucoup moins verbeux |
| Performances | Plus rapide (daemon + cache) | Beaucoup plus rapide |
| Multi-langages | ✅ Java + Kotlin + Scala | ✅ Java + Kotlin + Scala |
| Popularité Scala | ❌ Faible | N/A |
| Android | ✅ Standard | ❌ Non utilisé |

### Conseil final

> **Ne choisissez pas, maîtrisez les deux.**
>
> En entretien technique, un candidat capable de dire
> *"J'utilise SBT dans mes projets Scala, mais je suis à l'aise avec Maven
> dans un contexte d'équipe Java"* est nettement plus valorisé
> qu'un candidat qui ne connaît qu'un seul outil.
>
> Le code Scala est **identique** dans les deux cas — seul le fichier
> de configuration change. Passer de SBT à Maven ou l'inverse
> se fait en quelques heures une fois qu'on comprend les deux.
