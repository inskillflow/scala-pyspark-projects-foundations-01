## 1. Vue d'ensemble de l'environnement

Avant d'installer quoi que ce soit, comprenons **ce dont Spark a besoin** pour fonctionner :

```
┌─────────────────────────────────────────────────────────┐
│                    Votre programme Scala                 │
│                    (TP_StockSQL.scala)                   │
└───────────────────────────┬─────────────────────────────┘
                            │ compilé par
                            ▼
┌─────────────────────────────────────────────────────────┐
│              SBT (Scala Build Tool)                     │
│    - télécharge les dépendances (Spark, Scala...)       │
│    - compile le code Scala → bytecode JVM               │
│    - lance le programme                                 │
└───────────────────────────┬─────────────────────────────┘
                            │ s'exécute sur
                            ▼
┌─────────────────────────────────────────────────────────┐
│              Java Virtual Machine (JVM)                 │
│              Java JDK 21                                │
└─────────────────────────────────────────────────────────┘
```

**Ce qu'il faut installer :**

| Composant | Rôle | Version recommandée |
|---|---|---|
| **Java JDK** | Machine virtuelle qui exécute le code | 21 (LTS) |
| **SBT** | Outil de build Scala (comme Maven/Gradle) | 1.9.7 |
| **IntelliJ IDEA** | IDE (facultatif mais très utile) | 2024+ |

> Spark et Scala **ne s'installent pas séparément** — SBT les télécharge automatiquement
> depuis Internet en lisant le fichier `build.sbt` du projet.

## 2. Installation de Java JDK 21

### Pourquoi Java 21 ?

- C'est une version **LTS** (Long Term Support) → stable sur le long terme
- Spark 3.5.x la supporte officiellement
- IntelliJ 2024+ est optimisé pour Java 21

### Étape 1 — Télécharger le JDK

Rendez-vous sur : **https://www.oracle.com/java/technologies/downloads/#java21**

Choisir :
- Onglet **Windows**
- Fichier **x64 Installer** (`.exe`) — le plus simple

> **Alternative gratuite :** Microsoft OpenJDK 21
> https://learn.microsoft.com/fr-fr/java/openjdk/download

### Étape 2 — Installer

1. Double-cliquer sur le fichier `.exe` téléchargé
2. Cliquer **Next** → **Next** → **Close** (laisser les options par défaut)
3. Le JDK s'installe dans `C:\Program Files\Java\jdk-21\`

### Étape 3 — Configurer les variables d'environnement

Ouvrir **PowerShell en administrateur** et exécuter :

```powershell
# Définir JAVA_HOME
[System.Environment]::SetEnvironmentVariable(
    "JAVA_HOME",
    "C:\Program Files\Java\jdk-21",
    "Machine"
)

# Ajouter java au PATH
$currentPath = [System.Environment]::GetEnvironmentVariable("PATH", "Machine")
[System.Environment]::SetEnvironmentVariable(
    "PATH",
    "$currentPath;C:\Program Files\Java\jdk-21\bin",
    "Machine"
)
```

### Étape 4 — Vérifier l'installation

Fermer PowerShell, en ouvrir un nouveau, puis :

```powershell
java -version
```

Résultat attendu :
```
java version "21.0.3" 2024-04-16 LTS
Java(TM) SE Runtime Environment (build 21.0.3+7-LTS-152)
Java HotSpot(TM) 64-Bit Server VM (build 21.0.3+7-LTS-152, mixed mode, sharing)
```

> ❌ Si vous obtenez `'java' n'est pas reconnu` → fermer et rouvrir PowerShell,
> ou redémarrer Windows pour que le PATH soit pris en compte.

## 3. Installation de SBT sur Windows

### Qu'est-ce que SBT ?

SBT (*Scala Build Tool*) est l'équivalent de Maven (Java) ou npm (JavaScript) pour Scala.
Il lit le fichier `build.sbt` de votre projet et :
- Télécharge les bibliothèques (Spark, Scala, etc.) depuis Internet
- Compile votre code Scala
- Exécute votre programme

### Méthode 1 — Installateur MSI (recommandée pour débutants)

#### Étape 1 — Télécharger

Rendez-vous sur : **https://www.scala-sbt.org/download.html**

Cliquer sur **"Windows (msi)"** → téléchargement de `sbt-1.9.7.msi`

#### Étape 2 — Installer

1. Double-cliquer sur `sbt-1.9.7.msi`
2. Cliquer **Next** → **Next** → **Install**
3. Attendre la fin de l'installation
4. Cliquer **Finish**

L'installateur ajoute automatiquement SBT au PATH Windows.

#### Étape 3 — Vérifier

Ouvrir un **nouveau** PowerShell :

```powershell
sbt --version
```

Résultat attendu :
```
sbt version in this project: 1.9.7
sbt script version: 1.9.7
```

> ⏳ **La première exécution est lente !** SBT télécharge ses propres dépendances
> (~200 Mo). Laisser tourner jusqu'au prompt `sbt:>`.

---

### Méthode 2 — Via le sbt-launch.jar (si l'installateur ne fonctionne pas)

Si l'installateur MSI pose problème, cette méthode fonctionne toujours.

#### Étape 1 — Télécharger le JAR

```powershell
Invoke-WebRequest `
    -Uri "https://repo1.maven.org/maven2/org/scala-sbt/sbt-launch/1.9.7/sbt-launch-1.9.7.jar" `
    -OutFile "C:\tools\sbt-launch.jar"
```

#### Étape 2 — Créer un script de lancement

Créer le fichier `C:\tools\sbt.bat` avec ce contenu :

```batch
@echo off
java -jar "C:\tools\sbt-launch.jar" %*
```

#### Étape 3 — Ajouter C:\tools au PATH

```powershell
$currentPath = [System.Environment]::GetEnvironmentVariable("PATH", "Machine")
[System.Environment]::SetEnvironmentVariable(
    "PATH",
    "$currentPath;C:\tools",
    "Machine"
)
```

#### Étape 4 — Tester

```powershell
sbt --version
```

---

### Méthode 3 — Via Scoop (gestionnaire de paquets Windows)

[Scoop](https://scoop.sh/) est un gestionnaire de paquets léger pour Windows.

```powershell
# Installer Scoop (si pas encore installé)
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
Invoke-RestMethod -Uri https://get.scoop.sh | Invoke-Expression

# Installer SBT
scoop install sbt

# Vérifier
sbt --version
```

## 5. Configuration de JAVA_TOOL_OPTIONS

### Pourquoi cette variable est nécessaire ?

Spark 3.5.x accède à une classe interne de Java (`sun.nio.ch.DirectBuffer`)
pour gérer la mémoire. Depuis Java 17, ces classes internes sont **verrouillées** par défaut.

Sans cette variable, vous obtenez :
```
java.lang.IllegalAccessError: class org.apache.spark.storage.StorageUtils$
cannot access class sun.nio.ch.DirectBuffer
```

La variable `JAVA_TOOL_OPTIONS` est lue automatiquement par **toute JVM** au démarrage
— elle déverrouille l'accès au module nécessaire.

### Configuration (une seule fois, persistante)

**Via PowerShell :**

```powershell
[System.Environment]::SetEnvironmentVariable(
    "JAVA_TOOL_OPTIONS",
    "--add-opens=java.base/sun.nio.ch=ALL-UNNAMED",
    "User"
)
```

**Via l'interface Windows :**

```
Touche Windows → "Variables d'environnement"
→ "Modifier les variables d'environnement pour votre compte"
→ Nouvelle...
  Nom de la variable : JAVA_TOOL_OPTIONS
  Valeur             : --add-opens=java.base/sun.nio.ch=ALL-UNNAMED
→ OK → OK
```

**Vérification :**

Ouvrir un **nouveau** PowerShell :
```powershell
echo $env:JAVA_TOOL_OPTIONS
# Doit afficher : --add-opens=java.base/sun.nio.ch=ALL-UNNAMED
```

> ⚠️ **Important :** Après avoir défini cette variable, **fermer et relancer IntelliJ**.
> Les processus déjà ouverts ne voient pas les nouvelles variables d'environnement.

### Ce que vous verrez au démarrage (normal ✅)

```
Picked up JAVA_TOOL_OPTIONS: --add-opens=java.base/sun.nio.ch=ALL-UNNAMED
```

Cette ligne s'affiche à chaque exécution Java. C'est **normal** — elle confirme
que la variable a été prise en compte.

## 6. Exécution avec Docker (sans rien installer)

Docker permet de faire tourner votre programme dans un **conteneur isolé** qui contient
déjà Java, Scala et Spark — sans rien installer sur votre machine (sauf Docker lui-même).

### Schéma de fonctionnement

```
┌──────────────────── Votre machine Windows ─────────────────────┐
│                                                                  │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │          Conteneur Docker (Linux)                       │   │
│   │                                                         │   │
│   │   Java 21 + SBT + Spark 3.5.1                          │   │
│   │                    │                                    │   │
│   │         votre code Scala ◄──── monté depuis Windows    │   │
│   └─────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

### Étape 1 — Installer Docker Desktop

Rendez-vous sur : **https://www.docker.com/products/docker-desktop/**

1. Télécharger **Docker Desktop for Windows**
2. Lancer l'installateur → suivre les instructions
3. Redémarrer Windows
4. Lancer **Docker Desktop** depuis le bureau

Vérifier que Docker fonctionne :
```powershell
docker --version
# Docker version 26.x.x, build ...

docker run hello-world
# Hello from Docker!
```

### Étape 2 — Lancer les TPs avec Docker

Depuis le dossier du projet `projet1-scala` :

```powershell
# Se placer dans le dossier du projet
cd C:\00-projetsGA\projet1-scala

# Compiler et lancer TP_StockSQL dans un conteneur Spark
docker run --rm `
    -v "${PWD}:/app" `
    -w /app `
    --add-host=host.docker.internal:host-gateway `
    sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18 `
    sbt "runMain TP_StockSQL"
```

**Explication des options :**

| Option | Rôle |
|---|---|
| `--rm` | Supprime le conteneur après exécution (propre) |
| `-v "${PWD}:/app"` | Monte votre dossier projet dans le conteneur |
| `-w /app` | Définit `/app` comme répertoire de travail |
| `sbtscala/scala-sbt:...` | Image Docker avec Java 21 + SBT + Scala |
| `sbt "runMain TP_StockSQL"` | Commande à exécuter dans le conteneur |

> ⏳ **La première exécution télécharge l'image** (~500 Mo) — une seule fois.
> Les suivantes sont instantanées.

### Lancer TP_Income avec Docker

```powershell
docker run --rm `
    -v "${PWD}:/app" `
    -w /app `
    sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18 `
    sbt "runMain TP_Income"
```

### Mode interactif (shell SBT dans le conteneur)

```powershell
docker run --rm -it `
    -v "${PWD}:/app" `
    -w /app `
    sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18 `
    sbt
```

Vous obtenez un shell SBT interactif :
```
sbt:TP-SparkSQL> compile
sbt:TP-SparkSQL> runMain TP_StockSQL
sbt:TP-SparkSQL> runMain TP_Income
sbt:TP-SparkSQL> exit
```

## 7. Exécution avec Docker Compose

Docker Compose permet de décrire et lancer l'environnement dans un fichier de configuration,
ce qui simplifie et documente toutes les commandes.

### Étape 1 — Créer le fichier docker-compose.yml

Créer le fichier `docker-compose.yml` à la racine du projet :

```yaml
version: "3.9"

services:

  # ─────────────────────────────────────────────────────
  # Service 1 : Analyse des stocks AAPL (TP1 et TP3)
  # ─────────────────────────────────────────────────────
  tp-stocks:
    image: sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18
    container_name: spark-tp-stocks
    working_dir: /app
    volumes:
      - .:/app                          # monte tout le projet
      - sbt-cache:/root/.sbt            # cache SBT (évite de re-télécharger)
      - ivy-cache:/root/.ivy2           # cache des JARs
    environment:
      - JAVA_TOOL_OPTIONS=--add-opens=java.base/sun.nio.ch=ALL-UNNAMED
    command: sbt "runMain TP_StockSQL"

  # ─────────────────────────────────────────────────────
  # Service 2 : Analyse des revenus (TP Income)
  # ─────────────────────────────────────────────────────
  tp-income:
    image: sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18
    container_name: spark-tp-income
    working_dir: /app
    volumes:
      - .:/app
      - sbt-cache:/root/.sbt
      - ivy-cache:/root/.ivy2
    environment:
      - JAVA_TOOL_OPTIONS=--add-opens=java.base/sun.nio.ch=ALL-UNNAMED
    command: sbt "runMain TP_Income"

  # ─────────────────────────────────────────────────────
  # Service 3 : TP2 — Analyse multi-actions
  # ─────────────────────────────────────────────────────
  tp-multistocks:
    image: sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18
    container_name: spark-tp-multistocks
    working_dir: /app
    volumes:
      - .:/app
      - sbt-cache:/root/.sbt
      - ivy-cache:/root/.ivy2
    environment:
      - JAVA_TOOL_OPTIONS=--add-opens=java.base/sun.nio.ch=ALL-UNNAMED
    command: sbt "runMain Correction_TP2"

  # ─────────────────────────────────────────────────────
  # Service 4 : TP3 — Window Functions
  # ─────────────────────────────────────────────────────
  tp-window:
    image: sbtscala/scala-sbt:eclipse-temurin-21_sbt-1.9.7_scala-2.12.18
    container_name: spark-tp-window
    working_dir: /app
    volumes:
      - .:/app
      - sbt-cache:/root/.sbt
      - ivy-cache:/root/.ivy2
    environment:
      - JAVA_TOOL_OPTIONS=--add-opens=java.base/sun.nio.ch=ALL-UNNAMED
    command: sbt "runMain Correction_TP3"

# ─────────────────────────────────────────────────────
# Volumes nommés (cache persistant entre les runs)
# ─────────────────────────────────────────────────────
volumes:
  sbt-cache:
  ivy-cache:
```

### Étape 2 — Commandes Docker Compose

```powershell
# Se placer dans le dossier du projet
cd C:\00-projetsGA\projet1-scala

# Lancer uniquement TP_StockSQL
docker compose run --rm tp-stocks

# Lancer uniquement TP_Income
docker compose run --rm tp-income

# Lancer TP2 (multi-stocks)
docker compose run --rm tp-multistocks

# Lancer TP3 (window functions)
docker compose run --rm tp-window

# Lancer TOUS les TPs à la suite
docker compose up

# Nettoyer les conteneurs arrêtés
docker compose down

# Supprimer aussi les volumes (cache SBT) — repart de zéro
docker compose down -v
```

### Étape 3 — Comprendre les volumes de cache

```yaml
volumes:
  sbt-cache:/root/.sbt      # Cache des plugins SBT
  ivy-cache:/root/.ivy2     # Cache des JARs Spark (~500 Mo)
```

Sans ces volumes, chaque `docker compose run` re-téléchargerait tous les JARs.
Avec les volumes nommés, le cache est réutilisé entre les exécutions → **beaucoup plus rapide**.

### Vérifier les logs d'un service

```powershell
# Voir les logs du service tp-stocks
docker compose logs tp-stocks

# Suivre les logs en temps réel
docker compose logs -f tp-stocks
```

## 8. Troubleshooting — Erreurs fréquentes

---

### ❌ Erreur : `'java' n'est pas reconnu`

```
'java' n'est pas reconnu comme nom d'applet de commande
```

**Cause :** Java n'est pas dans le PATH.

**Solution :**
```powershell
# Vérifier si Java est installé
Test-Path "C:\Program Files\Java\jdk-21\bin\java.exe"

# Ajouter manuellement au PATH de la session courante
$env:PATH += ";C:\Program Files\Java\jdk-21\bin"

# Rendre permanent (nécessite redémarrage de PowerShell)
[System.Environment]::SetEnvironmentVariable(
    "PATH",
    [System.Environment]::GetEnvironmentVariable("PATH","Machine") + ";C:\Program Files\Java\jdk-21\bin",
    "Machine"
)
```

---

### ❌ Erreur : `'sbt' n'est pas reconnu`

**Cause :** SBT n'est pas installé ou pas dans le PATH.

**Solution rapide (sbt-launch.jar) :**
```powershell
# Télécharger sbt-launch.jar
Invoke-WebRequest `
    -Uri "https://repo1.maven.org/maven2/org/scala-sbt/sbt-launch/1.9.7/sbt-launch-1.9.7.jar" `
    -OutFile "C:\00-projetsGA\sbt-launch.jar"

# Lancer directement avec Java
$env:JAVA_TOOL_OPTIONS = "--add-opens=java.base/sun.nio.ch=ALL-UNNAMED"
cd C:\00-projetsGA\projet1-scala
& "C:\Program Files\Java\jdk-21\bin\java.exe" -jar "C:\00-projetsGA\sbt-launch.jar" "runMain TP_StockSQL"
```

---

### ❌ Erreur : `NoClassDefFoundError: org/apache/spark/rdd/RDD`

```
Error: Unable to initialize main class TP_StockSQL
Caused by: java.lang.NoClassDefFoundError: org/apache/spark/rdd/RDD
```

**Cause :** Les JARs Spark ne sont pas sur le classpath au moment de l'exécution.
Cela arrive quand les dépendances sont marquées `% "provided"` dans `build.sbt`.

**Solution :** Ouvrir `build.sbt` et retirer `% "provided"` :

```scala
// ❌ INCORRECT — Spark ne sera pas inclus à l'exécution
libraryDependencies ++= Seq(
  "org.apache.spark" %% "spark-core" % sparkVersion % "provided"
)

// ✅ CORRECT — Spark est inclus dans le classpath
libraryDependencies ++= Seq(
  "org.apache.spark" %% "spark-core" % sparkVersion
)
```

Après correction : clic droit sur `build.sbt` → **Reload SBT Project** dans IntelliJ.

---

### ❌ Erreur : `NoSuchMethodException: java.nio.DirectByteBuffer.<init>(long,int)`

```
Caused by: java.lang.NoSuchMethodException: java.nio.DirectByteBuffer.<init>(long,int)
```

**Cause :** Spark 3.3.x ou 3.4.x + Java 17/21 → incompatibilité de modules JVM.

**Solutions :**

**Option A — Passer à Spark 3.5.1 (recommandé) :**
```scala
// Dans build.sbt
val sparkVersion = "3.5.1"   // au lieu de "3.3.2"
```

**Option B — Ajouter JAVA_TOOL_OPTIONS :**
```powershell
[System.Environment]::SetEnvironmentVariable(
    "JAVA_TOOL_OPTIONS",
    "--add-opens=java.base/sun.nio.ch=ALL-UNNAMED",
    "User"
)
```

---

### ❌ Erreur : `IllegalAccessError: cannot access class sun.nio.ch.DirectBuffer`

```
java.lang.IllegalAccessError: class org.apache.spark.storage.StorageUtils$
cannot access class sun.nio.ch.DirectBuffer
```

**Cause :** `JAVA_TOOL_OPTIONS` n'est pas définie, ou IntelliJ a été lancé avant qu'elle soit définie.

**Diagnostic :**
```powershell
# Vérifier si la variable est définie
[System.Environment]::GetEnvironmentVariable("JAVA_TOOL_OPTIONS", "User")
# Doit afficher : --add-opens=java.base/sun.nio.ch=ALL-UNNAMED
```

**Solution :**
1. Définir la variable (voir section 5)
2. **Fermer complètement IntelliJ** (`File → Exit`)
3. Relancer IntelliJ
4. Vérifier que la console affiche `Picked up JAVA_TOOL_OPTIONS: ...`

---

### ❌ Erreur : `FileNotFoundException` (fichier CSV introuvable)

```
org.apache.hadoop.mapred.InvalidInputException:
Input path does not exist: file:/C:/00-projetsGA/projet1-scala/data/AAPL.csv
```

**Cause :** Le fichier CSV n'est pas au bon endroit.

**Diagnostic :**
```powershell
# Vérifier que le fichier existe
Test-Path "C:\00-projetsGA\projet1-scala\data\AAPL.csv"
# Doit afficher : True
```

**Solution :**
- Vérifier que le **Working Directory** dans IntelliJ est bien `$PROJECT_DIR$`
  (`Run → Edit Configurations → Working directory`)
- Ou utiliser un chemin absolu dans le code :
  ```scala
  val csvPath = "C:/00-projetsGA/projet1-scala/data/AAPL.csv"
  ```
  > Note : utiliser des `/` (slash) même sur Windows dans les chemins Spark/Hadoop.

### ❌ Avertissement : `WARN Utils: Service 'SparkUI' could not bind on port 4040`

```
WARN Utils: Service 'SparkUI' could not bind on port 4040. Attempting port 4041.
```

**Cause :** Le port 4040 est déjà utilisé (souvent par une autre instance Spark encore active).

**C'est normal et inoffensif ✅** — Spark utilise automatiquement le port suivant (4041).

Accéder au Spark UI : **http://localhost:4041**

Pour libérer le port si nécessaire :
```powershell
# Trouver le processus utilisant le port 4040
netstat -ano | findstr :4040

# Tuer le processus (remplacer XXXX par le PID)
Stop-Process -Id XXXX -Force
```

---

### ❌ Erreur : `OutOfMemoryError: Java heap space`

```
java.lang.OutOfMemoryError: Java heap space
```

**Cause :** Spark manque de mémoire heap (tas Java).

**Solution — Augmenter la mémoire dans `build.sbt` :**
```scala
// Dans build.sbt
javaOptions ++= Seq(
  "-Xms512m",     // mémoire initiale
  "-Xmx4g"        // mémoire maximale (adapter selon votre RAM)
)
fork := true
```

Ou dans la Run Configuration IntelliJ :
```
VM options : -Xms512m -Xmx4g
```

---

### ❌ Erreur : IntelliJ — `Cannot resolve symbol 'SparkSession'`

Les imports Spark apparaissent en rouge dans IntelliJ.

**Cause :** SBT n'a pas encore téléchargé les dépendances, ou le projet n'est pas correctement indexé.

**Solution :**
1. Attendre que la barre de progression en bas d'IntelliJ disparaisse
2. Si persistant : clic droit sur `build.sbt` → **Reload SBT Project**
3. Si toujours persistant : `File → Invalidate Caches → Invalidate and Restart`
